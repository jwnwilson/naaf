# A2 — NAAF UI (mock-data static SPA + full API contract) — Design

**Date:** 2026-06-29
**Status:** Approved design, pending implementation plan
**Milestone:** Phase A2 (board UI — expanded to the full design handoff)
**Design source:** `docs/design/` — `README.md` (the authoritative handoff: tokens, layout, all 7 screens, data model), `NAAF Hi-Fi.dc.html` (pixel reference), `NAAF Wireframes.dc.html` (structural reference)
**Backend it targets:** A1 control plane (Projects, WorkItems, Teams/AgentDefinitions) — see `docs/architecture.md`

## 1. Problem & goal

The A1 backend spine exists but has no UI. The design team delivered a complete, high-fidelity 7-screen product design (a Linear-style interface for managing AI agents against repos). Most of those screens depend on backends NAAF hasn't built yet (agent runs, live agents, chat, inbox, dashboards, token budgets → phases A3/A5/A6).

A2 builds the **entire UI now** as a **pixel-faithful, static Vite/React SPA driven by mock data**, and authors a **complete OpenAPI 3.1 API contract** that the mocks implement and a future backend builds against. This decouples UI progress from backend phases and makes the API contract the single source of truth.

### Success criterion

> `pnpm dev` (and a static `pnpm build`) renders all 7 screens — Dashboard, Inbox, Projects (Board + Issues list), Work-item Detail (Spec/Attachments/Activity/Agent/Subagents), Agent Monitor, Settings — faithful to `NAAF Hi-Fi`, fully navigable, with the app shell (sidebar / topbar / collapsible chat panel), every response served by MSW from typed fixtures. **Zero backend dependency.** Swapping to the real API (a later slice, A2-4) is: disable MSW, point the client at `/api`.

## 2. Decisions log

| Decision | Choice | Rationale |
|---|---|---|
| Fidelity & data | **Full 7-screen design, mock-data static SPA** | Realizes the whole design now; no dependency on unbuilt backends |
| Contract | **OpenAPI 3.1 doc is the single source of truth** | Language-neutral; future Python backend implements it; generates TS types |
| Mock seam | **MSW (Mock Service Worker)** intercepts real `fetch` → fixtures | Components make genuine HTTP today; live swap = flip MSW off |
| Types | `openapi-typescript` generates `schema.d.ts` from the OpenAPI doc | One source of truth for types |
| Envelope | Reuse A1's `{success, data, error}` (+`meta`) | Consistency with the existing backend |
| Server-state | **React Query**; SSE via a small `EventSource` hook | Caching/keys/polling; streaming run logs + chat |
| Stack | Vite + React + TypeScript + Tailwind + React Router; **pnpm** | Per `CLAUDE.md` (`projects/ui`, pnpm, never npm) |
| Tests | vitest + React Testing Library; MSW reused in node | Per `CLAUDE.md` (`pnpm test` = vitest) |

## 3. Scope

**In scope:** the full SPA (all 7 screens + shell + chat panel chrome), the design system (tokens + primitives + icons), the typed mock-data layer (OpenAPI doc, generated types, client, React Query hooks, MSW handlers + fixtures + mock SSE streamers), routing, loading/empty/error states, and the `pnpm lint`/`test`/`build` gates.

**Out of scope (explicit follow-ups):**
- **A2-4 live-API swap:** wiring Projects/WorkItems/Teams to the real A1 API (disable MSW for those paths, reconcile statuses — see §4).
- Real backends for runs/chat/inbox/dashboard/budget/agents (those are A3/A5/A6; A2 only mocks them, against the contract).
- Auth UI (A1 is dev-auth; the SPA assumes the single dev user). Real-time transport hardening (WebSocket) — A2 uses SSE per the contract.

## 4. The API contract (OpenAPI 3.1)

Authored at `projects/ui/openapi/naaf-api.yaml`. Every response is the envelope `{success, data, error}`; list endpoints add `meta: {total, page_size, page_number}`. Resources (derived from the handoff's data model + every screen's needs):

| Group | Endpoints (representative) | Drives |
|---|---|---|
| Projects | `GET/POST /projects`, `GET/PATCH/DELETE /projects/{id}` (incl. `itemCount`, `repoUrl`) | sidebar project list, board header |
| Work items | `GET /projects/{id}/board` (epic→feature→task tree), `GET /work-items` (filters: `status`, `epic`, `project`, pagination), `GET /work-items/{id}`, `POST /projects/{id}/work-items`, `PATCH /work-items/{id}`, `POST /work-items/{id}/transition` | Board, Issues list, Detail |
| Teams/Agents | `GET /teams`, `GET /agent-definitions`, `PATCH /agent-definitions/{id}` (settings: model/token-limit/system-prompt/enabled), `GET /agents` (live: status running/idle/paused, model, `currentItemId`, `progress`, token usage) | Agents ribbon, Settings |
| Runs | `GET /runs`, `GET /runs/{id}` (steps[1..6], logLines, tokenUsage, cost, status), `GET /work-items/{id}/run` (active run or null), **SSE** `GET /runs/{id}/stream` (incremental log lines + step transitions) | Agent Monitor (D2), ribbon |
| Inbox | `GET /inbox` (filter by type, unread count), `GET /inbox/{id}` (conversation), `POST /inbox/{id}/read`, `POST /inbox/mark-all-read` | Inbox, sidebar unread badge |
| Chat | `GET /threads`, `GET /threads/{id}/messages`, `POST /threads/{id}/messages`, **SSE** stream for agent replies/typing | Chat panel, Inbox conversation |
| Dashboard | `GET /dashboard/metrics` (active agents, spend, tokens, counts), `GET /dashboard/token-usage` (daily series for bar chart), `GET /activity` (feed) | Dashboard |
| Budget | `GET /budget` (workspace token budget) | sidebar footer |

**Entity shapes** follow the handoff's TypeScript data model (`WorkItem`, `Agent`, `Project`, `InboxItem`, `Message`, `AgentRun`, `RunStep`, `LogLine`) — these become OpenAPI `components/schemas`.

**Status reconciliation (explicit follow-up, not a silent divergence):** the design's work-item statuses are `backlog · todo · in_progress · in_review · done`; the A1 backend enum is `to_do · in_progress · in_review · approved · done · blocked · failed`. The OpenAPI doc defines the **UI-canonical set** (`backlog, todo, in_progress, in_review, done`) as the target contract. A2-4 (live swap) must align the backend enum or provide a mapping layer. This is recorded in the spec and will be surfaced in the plan; A2 itself uses the UI-canonical set throughout the mock layer.

## 5. Repo structure (`projects/ui/`, pnpm)

```
projects/ui/
  package.json · pnpm-lock.yaml · vite.config.ts · tsconfig.json
  tailwind config · postcss config · index.html · .env (VITE_USE_MOCKS=true)
  openapi/naaf-api.yaml                 # OpenAPI 3.1 contract (source of truth)
  src/
    main.tsx                            # entry: start MSW (if mocks) then render
    app/                                # providers (QueryClient, Router), AppShell, ErrorBoundary, routes
      AppShell.tsx · Sidebar.tsx · Topbar.tsx · ChatPanel.tsx · routes.tsx
    modules/                            # feature slices — each owns components + hooks
      dashboard/ inbox/ board/ list/ detail/ agent-monitor/ settings/ chat/
    components/ui/                      # design-system primitives + Icon set
      Button.tsx · Chip.tsx · Tag.tsx · StatusBadge.tsx · StatusCircle.tsx
      PriorityBars.tsx · Avatar.tsx · Toggle.tsx · ProgressBar.tsx · PulseDot.tsx
      TypingIndicator.tsx · Card.tsx · MetricCard.tsx · KanbanCard.tsx · ListRow.tsx
      icons/ (≈14 inline SVG icon components, currentColor)
    lib/
      api/
        schema.d.ts                     # generated by `pnpm gen:api`
        client.ts                       # envelope-aware typed fetch + ApiError
        queryKeys.ts                    # React Query key factories
        hooks/                          # useBoard, useWorkItem, useInbox, useRun(SSE), useThreads, useDashboard, useAgents, useBudget, useSettings, useAgentDefinitions
        mocks/
          browser.ts · server.ts        # MSW setup (worker / node)
          handlers.ts                   # implement every OpenAPI path
          sse.ts                        # mock run-log + chat streamers
          fixtures/                     # seed data across all states
      theme/ tokens.css                 # design tokens as CSS variables
      hooks/ useLocalStorage.ts · useEventSource.ts
    test/ setup.ts                      # vitest + RTL + MSW node setup
```

Placement: design-system primitives in `components/ui/`; feature logic in `modules/<slice>/`; all server-state + mock data in `lib/api/`; tokens/theme in `lib/theme/`. Matches the `CLAUDE.md` UI layout (app / modules / components/ui / lib/api).

## 6. Design system

- **Tokens (`lib/theme/tokens.css`):** the handoff's exact values as CSS variables — backgrounds (`--bg-base:#0e0f11` … `--bg-inset`), text (`--text-1..7`), borders (subtle/normal/strong), accent violet (`--accent:#7c6cf0` + bg/hover/border/text), semantic status colors (green/red/yellow/blue), and the `pulse` keyframes. Tailwind is configured to consume these tokens (theme colors + a custom type scale covering the 9–17px sizes, 4px-base spacing, 3–9px radii, the frame shadow). Pixel-exact values come from the token theme; rare one-offs use Tailwind arbitrary values.
- **Primitives (`components/ui/`):** small, token-driven, independently testable —
  - `Icon` set (≈14 inline SVGs, `currentColor`): dashboard, inbox, projects/folder, agents/chip, settings/gear, search, git-repo, list, grid, plus, chevron-down, chevron-right, check, document.
  - `StatusCircle` — SVG `backlog/todo/in_progress/in_review/done` using the exact circle math (r=4.5, circumference ≈28.27, half=14.14, three-quarter=21.2) and colors from the handoff.
  - `PriorityBars` (3 stepped bars), `Avatar` (user circle / agent rounded, mono initials), `Button` (primary/secondary/tertiary), `Chip`/`FilterChip`, `Tag` (epic), `StatusBadge` (inbox action_needed/review_needed/info/resolved + RUNNING/IDLE), `Toggle` (26×14 switch), `ProgressBar`, `PulseDot`, `TypingIndicator`, `Card`/`MetricCard`, `KanbanCard`, `ListRow`.

## 7. Screens → routes (React Router)

The **app shell** (`app/AppShell`) renders Sidebar (workspace header, search, nav with badges, project list, token-budget footer) / Topbar (title, count, filters, view switcher, New) / collapsible **ChatPanel** (open ↔ 34px strip; open/closed persisted in `localStorage`) around every route.

| Route | Module | Screen (handoff) |
|---|---|---|
| `/` → redirect `/projects?view=board` | — | default = Board |
| `/dashboard` | `dashboard` | E — metric cards, running-agents panel, token bar chart, activity feed |
| `/inbox` (+ `/inbox/:id`) | `inbox` | G — notification list (356px) + conversation pane, badges, quick actions |
| `/projects?view=board\|list` | `board` / `list` | B — live-agents ribbon (76px) + 5 kanban columns / A — grouped issue rows |
| `/projects/:projectId/items/:itemId` | `detail` | D — breadcrumb, item header, tabs: Spec · Attachments · Activity · Agent · Subagents |
| ↳ Agent tab | `agent-monitor` | D2 — step timeline (6 steps), log stream, token meter |
| `/settings/agents` | `settings` | F — settings subnav, lead-agent card, subagents table |

Behaviors from the handoff: list/board toggle (URL `view` param); in-progress kanban column highlight; kanban card → detail; agent name in ribbon → detail with Agent tab; inbox item → conversation; CSS-driven pulse animations (running dot 2s, RUNNING badge 3s, typing dots staggered); chat thread tabs switch active thread; Detail default tab = Spec, Agent tab dot pulses on active run.

## 8. Data layer & state

- `pnpm gen:api` → `openapi-typescript openapi/naaf-api.yaml -o src/lib/api/schema.d.ts`.
- `client.ts`: typed `fetch` wrapper that unwraps the envelope (returns `data`, throws typed `ApiError` on `success:false` or network/HTTP error); base URL `/api`, mocks toggled by `VITE_USE_MOCKS` (default `true`).
- `queryKeys.ts` + `hooks/`: one React Query hook per resource; `useRun`/chat use a `useEventSource` hook for SSE.
- `mocks/`: MSW handlers implementing **every** OpenAPI path over in-memory fixtures; `sse.ts` streams mock run-log lines + chat typing; `fixtures/` seed data spanning every visual state (statuses incl. backlog/done, running + idle + paused agents, a run with completed/active/pending steps and log lines, all four inbox badge types incl. resolved, threads with agent/user messages, dashboard metrics + a daily token series, a budget near a threshold, lead + subagent settings).
- **Non-server state:** chat panel open/closed (`localStorage`), active board view (URL `view`), active chat thread (in-memory), inbox selected item (route param). A small `useLocalStorage` hook backs persistence.

## 9. Testing & gates

- **vitest + React Testing Library**; MSW reused in node (`test/setup.ts`) for hook/integration tests.
- Behavior-focused tests (not pixel diffs): routing + nav active states; board grouping-by-status and list/board toggle; `StatusCircle` renders the correct variant per status; ChatPanel collapse + localStorage persistence; inbox filter tabs + unread count; settings `Toggle`; `client` envelope-unwrap + error path; `useEventSource`/`useRun` consuming streamed lines; sidebar badges reflect fixture counts.
- **Gates:** `pnpm lint` (eslint + `tsc --noEmit`), `pnpm test` (vitest), `pnpm build` (`tsc -b && vite build`) all green. Coverage focuses on `lib/` (client, hooks, mock handlers, utils) and module logic; presentational pixel components get render-smoke tests rather than a blanket percentage on markup.

## 10. Build / static output & error handling

- `pnpm build` → static `dist/` with MSW bundled = a fully offline, deployable demo of all 7 screens (the "static site"). Dev server (`pnpm dev`) runs MSW and proxies `/api` → `:8000` (dormant until the A2-4 live swap).
- The handoff is happy-path; the SPA still provides minimal **loading** skeletons, **empty** states, and **error** surfaces — an app-shell `ErrorBoundary` plus per-query error/loading states on Board, Inbox, and Detail — because even mocked HTTP can fail and the live swap will introduce real failures.

## 11. Implementation note (for the plan)

This spec is intentionally one document covering the whole UI + contract (per the decision to keep A2 cohesive). The **implementation plan** should still phase the work so each phase is independently reviewable, in this order: (1) scaffold + tokens + design-system primitives + icons; (2) OpenAPI contract + generated types + client + MSW + fixtures; (3) app shell + routing + sidebar/topbar/chat chrome; (4) Board + Issues list; (5) Work-item Detail + Agent Monitor; (6) Inbox; (7) Dashboard; (8) Settings; (9) chat panel content + final gate. Each phase ends green on `pnpm lint`/`test`/`build`.
